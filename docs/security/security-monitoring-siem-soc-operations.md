# 3.6 Security Monitoring & SIEM/SOC Operations

## 1. Log aggregation & correlation architecture

### Technical Definition
Log aggregation and correlation architecture is the foundational infrastructure of a Security Information and Event Management (SIEM) system, designed to ingest, normalize, and analyze massive volumes of security telemetry from disparate sources across the enterprise. Aggregation involves the collection of raw logs from endpoints, network devices, cloud services, and applications into a centralized data lake or messaging bus. Correlation is the subsequent process of applying logic to these normalized events to identify patterns, anomalies, or sequences of activity that indicate a security incident, transforming raw data into actionable intelligence.

### Underlying Mechanism
The architecture typically employs a tiered ingestion pipeline. At the edge, log forwarders—such as Syslog-ng, Fluentd, or Windows Event Forwarding (WEF/WEC)—collect raw telemetry and transmit it to a centralized messaging queue, such as Apache Kafka or Azure Event Hubs, which acts as a buffer to handle high-volume bursts. From the queue, data is ingested into a normalization layer where it is parsed into a common schema, such as the Advanced Security Information Model (ASIM) or the Open Source Security Events Metadata (OSSEM), ensuring that fields like source IP, user identity, and timestamp are consistent across all sources. The correlation engine then processes this normalized stream, utilizing in-memory indexing structures to cross-reference events in real-time. This engine links identity-based signals, which provide the context for user-centric threats as detailed in Section 3.2, with the host-level telemetry and forensic artifacts generated by the EDR/XDR systems described in Section 3.4. By correlating these disparate data points, the architecture can reconstruct complex attack chains that would be invisible if analyzed in isolation.

### Why It Exists
Historically, security logs were siloed within individual systems, making it impossible to detect multi-stage attacks that spanned different infrastructure layers. As enterprises grew in complexity and scale, the volume of logs exceeded the capacity of manual analysis, creating a massive visibility gap. Log aggregation and correlation architecture exists to solve this by centralizing telemetry, providing the scale required to ingest petabytes of data, and enabling the automated detection of threats that rely on the subtle interplay between different system components.

### Enterprise / Banking Reality
In Tier-1 banking, the log aggregation architecture must be designed for extreme scale, high availability, and strict regulatory compliance. Banks must adhere to mandates such as PCI-DSS, which requires comprehensive log retention and integrity, and FFIEC guidelines that demand real-time monitoring of critical systems. The architecture must support multi-tenancy, allowing for the logical isolation of sensitive environments like SWIFT or trading platforms, while providing a unified view for the SOC. Furthermore, the system must be resilient to data loss, ensuring that even during a massive DDoS attack or infrastructure failure, security logs are preserved and available for forensic analysis.

### Operational Considerations
Operationalizing this architecture requires a rigorous approach to pipeline health and data quality. Administrators must monitor the ingestion rate, latency, and error rates of the log forwarders and the messaging queue. Normalization is a continuous effort; as new log sources are added, they must be mapped to the common schema to ensure they are usable by the correlation engine. Monitoring the health of the SIEM itself is critical; administrators must track indexing performance, query latency, and storage utilization.

### Common Misconceptions
!!! warning
    A common misconception is that "more data is better" and that ingesting every possible log source will automatically improve security. In reality, ingesting low-value logs increases storage costs, degrades query performance, and creates "noise" that obscures critical alerts. Another error is assuming that the SIEM is a "magic box" that will automatically detect all threats; it requires careful tuning, rule development, and a deep understanding of the environment to be effective.

### Interview Angle
1. Question: How do you architect a log aggregation pipeline to handle the scale of a global banking enterprise while maintaining cost-efficiency?
   Answer: We implement a tiered storage strategy. High-value, high-fidelity logs are ingested into the SIEM for real-time correlation and alerting. Lower-value, high-volume logs are sent directly to a low-cost data lake for long-term retention and retrospective hunting. We also use data filtering and aggregation at the edge to reduce the volume of data sent to the SIEM, ensuring that we only pay for the telemetry that provides actionable security value.
2. Question: What are the architectural trade-offs between agent-based and agentless log collection in a high-security environment?
   Answer: Agent-based collection provides more reliable, structured data and allows for local filtering, which reduces bandwidth usage. Agentless collection, such as Syslog or WEF, is easier to deploy but can be less reliable and harder to manage at scale. In a Tier-1 bank, we use a hybrid approach: agent-based collection for critical endpoints and servers, and agentless collection for network devices and legacy systems that cannot support agents.
3. Question: How do you ensure the integrity and non-repudiation of security logs in a banking environment?
   Answer: We implement end-to-end encryption for log transmission and store logs in an immutable, write-once-read-many (WORM) storage environment. We also use digital signatures to verify the integrity of the logs at each stage of the pipeline, ensuring that they have not been tampered with from the moment they are generated until they are archived.
4. Question: How do you manage the schema evolution of log sources in a large, complex environment?
   Answer: We implement a centralized schema management process. Any changes to log formats must be reviewed and approved by the security engineering team, and the normalization logic must be updated and tested before the changes are deployed to production. We also use automated testing to ensure that new log formats do not break existing correlation rules.
